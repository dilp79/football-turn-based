# Layout Wireframes

Применение skill `game-layout-wireframes` к проекту "Пошаговый футбол" на основе [prd.md.md](../prd.md.md) и текущих дизайн-заметок.

### Layout Count

Нужно `3` distinct layouts, потому что у игры три разные первичные задачи: экономическая сборка состава, пространственная расстановка на поле и живая тактическая игра на ходу. Подсостояния матча вроде выбора сектора удара или рикошета не требуют нового layout, им достаточно overlay внутри match layout.

### Layout Map

- `Squad Builder Layout` -> used for `драфт состава`, `покупка карточек`, `проверка бюджета и позиционного состава`
- `Formation Placement Layout` -> used for `предматчевая расстановка игроков`, `поочерёдная постановка пятёрок на своей половине`
- `Live Match Layout` -> used for `ходы во время матча`, `выбор действия`, `перехват`, `дриблинг`, `подкат`, `выбор сектора удара`, `выбор прыжка вратаря`, `выбор клетки рикошета`, `конец матча`

### Layout Specs

#### Squad Builder Layout

- `Used when`: с момента выдачи 100 монет каждому игроку до момента, когда оба игрока собрали легальные составы по 5 человек
- `Primary task`: купить пятёрку под бюджет и позиционные ограничения
- `Dominant area`: рынок карточек игроков, `68-74%` ширины экрана; это основной рабочий стол режима
- `Secondary areas`: правая колонка состава активного игрока `18-22%`; верхняя полоска статуса драфта `8-10%`; небольшая зона состава соперника или счётчика прогресса `6-10%`
- `Components`: верхний hot-seat статус с именем активного игрока и остатком монет; сетка доступных карточек игроков; компактный блок правил драфта `5 игроков / 1 GK / 100 монет`; колонка текущего состава активного игрока; колонка прогресса второго игрока
- `Hidden or collapsed`: поле 15x9; мяч; все матчевые action buttons; кубики; очки действий; лог эпизодов; выбор клетки; overlays ударов
- `Player interaction`: игрок сначала смотрит на бюджет и незакрытые слоты состава; затем сканирует рынок карточек; кликом открывает/выбирает карточку; сверяет цену, позицию и статы; подтверждает покупку; состав и бюджет обновляются сразу; ход покупки переходит ко второму игроку; цикл повторяется, пока оба состава не станут легальными
- `Why this cannot share the match layout`: здесь нет пространственной задачи на поле, поэтому показ поля только создаёт шум и отнимает место у карточек, которые в этом режиме и есть главный объект взаимодействия

#### Formation Placement Layout

- `Used when`: сразу после завершения драфта и стартового броска до тех пор, пока оба игрока не расставят свои пятёрки
- `Primary task`: быстро и безошибочно поставить выбранных игроков в допустимые клетки своей половины
- `Dominant area`: поле 15x9 с крупными клетками и явной подсветкой валидных зон, `62-70%` ширины экрана
- `Secondary areas`: боковая колонка списка ещё не выставленных игроков `18-22%`; верхняя статусная панель `8-10%`; компактный блок подсказок и ограничений `8-12%`
- `Components`: поле с линией центра и явным обозначением ворот; подсветка доступных клеток для выбранного игрока; список выбранной пятёрки с состояниями `не выставлен / уже на поле`; индикатор активной команды; карточка выбранного игрока; правило для вратаря `обязан стоять в воротах`; кнопка подтверждения готовности или авто-переход после постановки всех игроков
- `Hidden or collapsed`: рынок карточек; полный каталог футболистов; live action panel; кубики и очки действий; крупный лог; матчевые супер-оверлеи
- `Player interaction`: игрок видит поле первым; выбирает из боковой колонки одного из невыставленных футболистов; допустимые клетки загораются на поле; кликом по клетке ставит игрока; список справа обновляется; система автоматически переключает фокус на следующего невыставленного игрока; после завершения пятёрки управление передаётся второму игроку; после второй расстановки матч стартует
- `Why this cannot share the match layout`: хотя поле уже нужно, логика режима не про выбор тактических действий, а про одноразовое legal placement; матчевые кнопки и live HUD здесь только перегружают экран

#### Live Match Layout

- `Used when`: с первого хода после расстановки до конца матча, включая реактивные подсостояния внутри хода
- `Primary task`: читать тактическую ситуацию и тратить очки действий на оптимальную последовательность действий
- `Dominant area`: игровое поле с фишками, мячом и подсветкой целевых клеток, `56-64%` ширины экрана
- `Secondary areas`: правая командная колонка с выбранным игроком и action controls `20-24%`; верхняя компактная полоса счёта, активной команды, кубиков и ОД `10-12%`; нижняя или боковая вспомогательная зона статуса составов и редкого лога `10-14%`
- `Components`: поле 15x9; визуальные маркеры команд, вратаря, выбранного игрока, носителя мяча и свободного мяча; верхний turn bar со счётом, активным игроком, двумя кубиками и очками действий; контекстная карточка выбранного футболиста; action buttons `перемещение`, `пас`, `удар`, `отбор`, `подкат`, `дриблинг`; подсветка допустимых целей на поле; overlay для выбора сектора удара; overlay для выбора прыжка вратаря; overlay для выбора клетки рикошета
- `Hidden or collapsed`: рынок карточек; полноразмерные драфтовые карточки всех 20 игроков; placement-инструкции; большие постоянные панели, не влияющие на текущий ход
- `Player interaction`: игрок сначала считывает верхнюю панель хода и владения; затем кликом выбирает своего футболиста на поле; правая колонка переключается на доступные действия для этой фишки; игрок кликает действие; допустимые клетки или цели подсвечиваются прямо на поле; второй клик по клетке или по сектору завершает эпизод; поле, мяч, ОД и контекстная карточка обновляются сразу; цикл повторяется до завершения хода

### Shared Components

- Верхняя hot-seat шапка с названием игры, фазой и идентичностью двух команд может сохранять одинаковое место во всех layout’ах
- Компактный phase/status bar может жить в верхней части всех layout’ов, но его содержимое должно меняться по режиму
- Карточка выбранного игрока может сохранять одинаковую позицию в правой колонке в `Formation Placement Layout` и `Live Match Layout`
- Центрированный modal/overlay слой для подтверждений, handoff и реактивных выборов можно держать единым для всех layout’ов

### Transition Rules

- `Squad Builder Layout -> Formation Placement Layout`: переход происходит, когда оба игрока собрали легальные составы `5 игроков`, включая `1 GK`; должны сохраниться купленные карточки, остатки монет, порядок первого хода и составы команд
- `Formation Placement Layout -> Live Match Layout`: переход происходит, когда обе команды расставили всех своих игроков по допустимым клеткам; должны сохраниться координаты игроков, стартующий игрок, роли, суперспособности и состояние мяча
- `Live Match Layout` не должен распадаться на новые layout’ы для каждого эпизода; `shot aim`, `keeper dive`, `bounce choice` и похожие состояния должны жить как overlays поверх того же match layout
- `Live Match Layout -> End State`: в текущем объёме проекта отдельный results layout не обязателен; после победы достаточно end overlay поверх match layout с итоговым счётом и действиями `новый матч / реванш`

### Risks

- Пытаться решить все три режима одним универсальным layout и получить экран, в котором рынок карточек, поле и action panel конкурируют одновременно
- Показывать поле во время драфта и тем самым забирать площадь у главного объекта режима, то есть карточек игроков
- Делать расстановку как почти тот же матчевый экран, из-за чего игроку приходится игнорировать ненужные live controls
- Оставлять в матче полноразмерные карточки всех игроков вместо компактного контекстного блока выбранной фишки
- Давать одинаковый визуальный вес полю, логу, правой колонке и верхнему статусу, хотя в live режиме поле должно побеждать почти всегда
- Не зафиксировать, что реактивные эпизоды матча решаются overlays, а не отдельными layout-переходами
